这道题（CSES - Graph Paths II）非常经典，它将图论中的路径问题与线性代数中的矩阵运算巧妙地结合在了一起。在研究现代密码学（如ZKP、FHE等）底层依赖的抽象代数时，我们经常会探讨不同的环（Ring）和半环（Semiring）结构，而这道题的核心恰恰是将传统的矩阵运算映射到了一个被称为**热带半环（Tropical Semiring）**或**最小加法代数（Min-Plus Algebra）**的数学结构上。

### 一、 为什么图论能映射到矩阵运算？（数学原理与推导）

要理解这一点，我们需要先回顾**普通矩阵乘法**在图论中的物理意义，然后再看如何将其修改为求**最短路径**。

#### 1. 普通矩阵乘法的图论意义（求路径条数）
假设我们有一个无权有向图的邻接矩阵 $A$：
* 如果节点 $i$ 到 $j$ 有边，则 $A_{i,j} = 1$。
* 如果没有边，则 $A_{i,j} = 0$。

如果我们把矩阵 $A$ 自己乘自己，得到 $C = A \times A$（即 $A^2$），按照普通矩阵乘法公式：

$$
C_{i,j} = \sum_{k=1}^n (A_{i,k} \times A_{k,j})
$$

这个公式在物理上意味着什么？
* $A_{i,k} \times A_{k,j}$ 的结果只有在“$i$ 到 $k$ 有边”且“$k$ 到 $j$ 有边”时才为 1，否则为 0。
* 外层的 $\sum$（求和）则是在枚举所有的中间节点 $k$。
* 因此，$C_{i,j}$ 的值代表了：**从节点 $i$ 经过恰好 2 条边到达节点 $j$ 的路径总数**。

同理，若求 $A^k$，矩阵中第 $i$ 行第 $j$ 列的值，就是从 $i$ 到 $j$ 恰好经过 $k$ 条边的路径总数。

#### 2. “最小加”代数：从求“条数”到求“最短路径”
现在，我们的目标变了：不再是求路径的“条数”，而是求恰好经过 $k$ 条边的**最小权重和（最短路径）**。



我们需要对普通矩阵乘法的公式进行“降级替换”（映射到热带半环）：
* **将乘法（$\times$）替换为加法（$+$）**：在路径中，两段连续边的总代价是它们的权重之和。
* **将加法（$\sum$）替换为取最小值（$\min$）**：我们不关心有多少条路，我们只关心所有可能的中间节点 $k$ 中，哪一条带来的总代价最小。

由此，我们得到了新的矩阵乘法公式：

$$
C_{i,j} = \min_{k=1}^n (A_{i,k} + B_{k,j})
$$

在这个新的运算体系下：
* 矩阵 $A$ 的初始状态：$A_{i,j}$ 存储 $i$ 到 $j$ 的边权。如果没边，设为正无穷（$\infty$）。
* $C = A \otimes A$（这里的 $\otimes$ 代表新的矩阵乘法）：$C_{i,j}$ 就代表了**从节点 $i$ 经过恰好 2 条边到达节点 $j$ 的最短路径长度**。
* 顺理成章地，计算 $A^k$（即 $A$ 连乘 $k$ 次），得到的矩阵中的元素 $A^k_{i,j}$ 就是从 $i$ 到 $j$ 恰好经过 $k$ 条边的最短路径。

### 二、 矩阵快速幂加速（时间复杂度分析）

如果每次都做一次新的矩阵乘法，求 $A^k$ 需要做 $k-1$ 次矩阵乘法。每次矩阵乘法的时间复杂度是 $O(n^3)$。总时间复杂度将是 $O(k \cdot n^3)$。由于题目中 $k \le 10^9$，这绝对会超时。

但这里有一个关键的数学基石：**“最小加”运算体系同样满足结合律（Associativity）**。

$$
(A \otimes B) \otimes C = A \otimes (B \otimes C)
$$

只要满足结合律，我们就可以使用**矩阵快速幂**。
例如，求 $A^8$：
* 常规求法：$A \to A^2 \to A^3 \to \dots \to A^8$ （乘 7 次）
* 快速幂求法：$A \to A^2 \to A^4 \to A^8$ （只乘 3 次，每次都是前一个矩阵平方）

快速幂将乘法次数从 $k$ 降到了 $\log_2 k$。因此，最终的时间复杂度优化为 **$O(n^3 \log k)$**，完全可以满足题目要求。

### 三、 回归代码实现细节

两份代码的核心逻辑完全一致，都是实现了上述的“最小加矩阵快速幂”，但在工程实现上有一些区别。

#### 1. 关于“无穷大（INF）”的定义
由于运算中涉及加法 $A_{i,k} + B_{k,j}$，如果两边都是不存在的边（即两个无穷大相加），我们需要防止整数溢出（Overflow）。
* 代码 1 设定 `INF = 4e18`。
* 代码 2 设定 `INF = 2e18`。
在 `long long` 的上限约是 `9e18`，所以这两个 `INF` 的设定保证了两个无穷大相加不会变成负数（溢出）。

#### 2. 单位矩阵（Identity Matrix）的重定义
在普通矩阵乘法中，单位矩阵是对角线为 1，其余为 0。
在“最小加”半环中，单位矩阵 $E$ 必须满足 $A \otimes E = A$。
* 自己到自己的距离（0 条边）代价是 0，所以**对角线元素必须是 0**。
* 自己到别人的距离（0 条边）是不合法的，所以**非对角线元素必须是 INF**。
两份代码都正确实现了这个新的单位矩阵（如代码 1 中的 `res.m[i][i] = 0;` 和代码 2 的 `ident()` 函数）。

#### 3. 代码 1 与 代码 2 的差异对比

* **代码 1（迭代式快速幂 + 剪枝优化）：**
  采用了 `while(p > 0)` 的迭代写法，这是工业界和竞赛中最标准的快速幂写法。
  它做了一个非常棒的常数级优化（剪枝）：
  ```cpp
  if (m[i][k] == INF) continue; 
  ```
  因为如果 $m[i][k]$ 是无穷大，加上任何数还是无穷大，不可能成为最小值，直接跳过可以节省大量不必要的加法和 `min` 函数调用。这在图比较稀疏时能大幅提升运行速度。

* **代码 2（递归式快速幂 + 泛型模板）：**
  使用了 `template <int N>` 让矩阵大小在编译期确定，这有助于编译器做循环展开等底层优化。
  使用了递归式的快速幂：`return b & 1 ? a.power(b - 1) * a : (a * a).power(b / 2);`。数学上严谨，但在 C++ 中递归会带来函数调用栈的开销，且没有做上述的 `INF` 剪枝，常数性能上会略逊于代码 1。

#### 4. 重边处理
由于图论题目中可能存在两点之间有多条边，题目要求的是最小权重，因此在读入数据时，两份代码都做了去重保留最小值的处理：
`g.m[u][v] = min(g.m[u][v], c);`

### 总结
这道题的精髓在于**抽象代数的同构映射**。将原本用于求路径数量的（加法，乘法）环，平滑替换为了用于求最短路径的（取最小，加法）半环。因为这个新结构保留了结合律，所以可以无缝接入原本为普通矩阵设计的快速幂算法。这就是为什么求最短路径能和矩阵运算完美融合的本质原因。


对于致力于密码学（尤其是像ZKP、FHE这样依赖深层代数结构的领域）的同学来说，理解这种**代数结构的同构映射**（将图论问题映射到半环代数上）是非常核心的能力。

### 二、 核心运算：“最小加”矩阵乘法

这是整个算法的灵魂。普通矩阵乘法的公式是 $C_{i,j} = \sum_{k=1}^n (A_{i,k} \times B_{k,j})$。
在我们的“热带半环”中，乘法被替换成了公式：

$$
C_{i,j} = \min_{k=1}^n (A_{i,k} + B_{k,j})
$$

#### 1. 举例：手动计算 $G^2$ ($G \otimes G$)
$G^2$ 代表**恰好经过 2 条边**的最短路径。我们来计算 $G^2$ 中第 1 行第 3 列的值 $G^2_{1,3}$（即从节点 1 到节点 3 恰好经过 2 条边的最短距离）。

根据公式，我们需要遍历所有可能的中间节点 $k$ (1, 2, 3)：
* 当 $k=1$：$G_{1,1} + G_{1,3} = \infty + \infty = \infty$
* 当 $k=2$：$G_{1,2} + G_{2,3} = 5 + 4 = 9$ （代表路径 1 -> 2 -> 3）
* 当 $k=3$：$G_{1,3} + G_{3,3} = \infty + \infty = \infty$

取最小值：$\min(\infty, 9, \infty) = 9$。所以 $G^2_{1,3} = 9$。

如果我们将整个 $G^2$ 计算出来，矩阵如下：

| $G^2$ (2条边) | 节点 1 | 节点 2 | 节点 3 |
| :--- | :--- | :--- | :--- |
| **节点 1** | $\infty$ | $\infty$ | 9 |
| **节点 2** | 5 | 6 | $\infty$ |
| **节点 3** | $\infty$ | 6 | 6 |

*验证：矩阵中 $G^2_{2,1} = 5$，对应的路径是 2 -> 3 -> 1 (权重 4+1=5)，完全正确。*

#### 2. 映射到代码实现

两份代码都重载了 `operator*` 来实现这个公式，但**代码1**做了一个极其重要的工业级优化：

**代码1的实现（带剪枝）：**
```cpp
for (int i = 0; i < n; ++i) {
    for (int k = 0; k < n; ++k) {
        if (m[i][k] == INF) continue; // 【关键剪枝】：如果 A[i][k] 是无穷大，没必要再加 B[k][j] 了
        for (int j = 0; j < n; ++j) {
            if (b.m[k][j] == INF) continue; 
            res.m[i][j] = min(res.m[i][j], m[i][k] + b.m[k][j]); // 严格对应数学公式
        }
    }
}
```
* **严谨性批注**：代码1交换了内层循环的 $j$ 和 $k$ 的顺序。这种写法不仅方便了 `m[i][k] == INF` 的提早剪枝，还在计算机底层上提升了 CPU Cache 的命中率（因为按行连续访问内存），这是在系统工程中非常优秀的代码习惯。

**代码2的实现（朴素版）：**
```cpp
for (int i = 0; i < N; ++i) {
    for (int j = 0; j < N; ++j) {
        for (int k = 0; k < N; ++k) {
            r.m[i][j] = min(r.m[i][j], m[i][k] + b.m[k][j]);
        }
    }
}
```
* **严谨性批注**：代码2严格按照数学公式的书写顺序实现，没有做剪枝。在图较稀疏（$\infty$ 很多）时，会产生大量无意义的加法运算。

### 三、 矩阵快速幂与单位矩阵

题目要求 $k = 8$，也就是求 $G^8$。
利用结合律，我们不需要算 7 次乘法，而是：
$G^2 = G \otimes G$
$G^4 = G^2 \otimes G^2$
$G^8 = G^4 \otimes G^4$
只需 3 次矩阵乘法。

#### 1. 单位矩阵的定义
在快速幂中，我们需要一个“初始状态”或者说“底数”，即**单位矩阵 $E$**。
在普通加法乘法中，单位矩阵主对角线是 1，其余是 0。
在我们的“最小加”代数中，单位矩阵代表**经过 0 条边的最短距离**：
* 自己到自己（0条边）：距离是 0。
* 自己到别人（0条边）：不可能到达，距离是 $\infty$。

| 单位矩阵 $E$ | 节点 1 | 节点 2 | 节点 3 |
| :--- | :--- | :--- | :--- |
| **节点 1** | 0 | $\infty$ | $\infty$ |
| **节点 2** | $\infty$ | 0 | $\infty$ |
| **节点 3** | $\infty$ | $\infty$ | 0 |

#### 2. 映射到代码实现
**代码1的快速幂（迭代法）：**
```cpp
Matrix power(ll p) const {
    Matrix res(n);
    for (int i = 0; i < n; i++) res.m[i][i] = 0; // 构建单位矩阵
    
    Matrix a = *this;
    while (p > 0) {
        if (p & 1) res = res * a;
        a = a * a;
        p >>= 1;
    }
    return res;
}
```
* **客观评价**：迭代法是算法竞赛和底层库中最常用的实现方式，空间复杂度低，不会因为递归导致栈溢出。

**代码2的快速幂（递归法）：**
```cpp
static Matrix<N> ident() {
    Matrix<N> r(INF);
    for (int i = 0; i < N; i++) r.m[i][i] = 0; // 构建单位矩阵
    return r;
}

Matrix<N> power(ll b) {
    if (b == 0) return ident();
    auto a = *this;
    return b & 1 ? a.power(b - 1) * a : (a * a).power(b / 2);
}
```
* **客观评价**：递归法在数学表达上更加优雅，接近数学公式的定义。但每次递归都会产生一次 `Matrix<N>` 的值拷贝（`auto a = *this;`），在矩阵较大时，空间和时间的常数开销会比迭代法大。

# 没有乘法了 是不是就可以按照传统顺序了？
这是一个非常深刻的性能优化直觉。你提到的“没有乘法了”确实改变了指令级的开销（加法和 `min` 通常比浮点乘法快），但在计算机体系结构中，**$k$ 循环放在中间（即 $i \to k \to j$ 顺序）依然是性能最优的选择**。

这不仅仅是为了“剪枝”，更是为了压榨 **CPU 缓存（Cache）** 的效能。

---

### 1. 为什么 $k$ 在中间（$i, k, j$）比 $k$ 在最内层（$i, j, k$）快？

虽然数学公式通常写成 $C_{i,j} = \min_{k} (A_{i,k} + B_{k,j})$，对应 $k$ 在最内层，但代码实现中 $k$ 在中间（代码 1 的写法）具有压倒性优势：

#### **A. 内存访问的连续性（Spatial Locality）**
计算机读取内存不是一个字节一个字节读的，而是每次加载一整块（Cache Line，通常 64 字节）。
* **$k$ 在中间 ($i, k, j$)**：
    在最内层 $j$ 循环时，我们访问的是 `res.m[i][j]` 和 `b.m[k][j]`。注意，`j` 是数组的最后一维。这意味着随着 `j` 的增加，我们在内存中是**连续扫描**的。
* **$k$ 在最内层 ($i, j, k$)**：
    在最内层 $k$ 循环时，我们访问 `b.m[k][j]`。由于 `k` 是第一维，每次 $k+1$，内存地址都会跳跃一个整行（`MAXN * 8` 字节）。这会导致大量的 **Cache Miss（缓存缺失）**，CPU 必须频繁去主存（RAM）搬数据，速度会慢 10-100 倍。



#### **B. 逻辑剪枝的效率**
如你所见，代码 1 中：
```cpp
for (int i = 0; i < n; ++i) {
    for (int k = 0; k < n; ++k) {
        if (m[i][k] == INF) continue; // 就在这里！
        for (int j = 0; j < n; ++j) {
            res.m[i][j] = min(res.m[i][j], m[i][k] + b.m[k][j]);
        }
    }
}
```
如果 $k$ 在最内层，你必须进入最内层循环后才能判断 `m[i][k]` 是否为 `INF`。而 $k$ 在中间，一旦发现 `m[i][k] == INF`，可以直接**跳过整个 $j$ 循环（$n$ 次运算）**。这种“降维打击”式的剪枝在稀疏图或矩阵快速幂后期（INF 增多）时效果极其恐怖。

---

### 2. “没有乘法”对循环顺序的影响

你提到的“没有乘法”确实是一个关键点：
* 在**普通矩阵乘法**中，最内层是 `res[i][j] += a[i][k] * b[k][j]`。编译器通常可以利用 FMA（乘加指令）和向量化（SIMD）来加速。
* 在**最小加代数**中，最内层是 `res[i][j] = min(res[i][j], a[i][k] + b[k][j])`。

虽然没有了乘法，但 `min` 和 `+` 同样可以被现代编译器的 **SIMD（单指令多数据流）** 优化。当 $j$ 在最内层连续访问时，编译器可以使用类似 `pminsd`（并行比较整数）和 `paddd`（并行加法）的指令，一次性处理 4 个或 8 个元素的 `min-plus` 运算。如果 $k$ 在内层导致内存访问不连续，这种硬件级的并行加速就失效了。

---

### 3. 重点总结：三层循环的最优排列

在矩阵运算（无论是普通乘法还是最小加）中，循环顺序的“金科玉律”是：**让最内层的变量是矩阵在内存中连续存储的那一维。**

对于 C++/C 这种行优先（Row-major）存储的语言：
* **最优顺序**：$i, k, j$ （$j$ 变动最快，内存连续）。
* **次优顺序**：$k, i, j$ （$j$ 变动最快，内存连续）。
* **最差顺序**：$i, j, k$ （$k$ 变动最快，内存跨行跳转，且无法有效剪枝 $m[i][k]$）。

### 4. 结合你的密码学背景（ZKP/FHE）
在处理 ZKP（如卷积或大整数矩阵）或 FHE 的密文矩阵运算时，内存布局（Memory Layout）和 Cache 命中率对性能的影响往往超过了算法本身的复杂度。你会发现，虽然数学公式写得天花乱坠，但在底层实现时，为了性能，我们经常会进行“矩阵转置”或者调整循环顺序，其背后的数学依据就是**局部性原理**。

**所以结论是：** 虽然乘法没了，但为了 CPU 缓存的幸福和剪枝的快乐，**$k$ 依然必须放在中间**。
